**الإصدار:** 1.0  
**الحالة:** مرجع تنفيذي معتمد  
**النطاق:** المنتج، المحتوى، التقنية، الأمان، البيانات، SEO/GEO، والوكلاء الذكيون

## 1. الميثاق التشغيلي

### المهمة

إرساء معيار عربي متقدم للصحافة الاستقصائية الرقمية عبر دمج الدقة الصحفية، التحقق الصارم، OSINT، تحليل البيانات، والذكاء الاصطناعي المسؤول.

### القيم الجوهرية

- **الحياد المنهجي:** فصل الأدلة عن الرأي، والوقائع عن الاستنتاج.
- **التحقق الصارم:** لا نشر قبل اجتياز مسار تحقق موثق.
- **شفافية المصادر:** توثيق المصدر، تاريخ الوصول، النسخة المؤرشفة، وسياق الاستخدام.
- **قابلية التكرار:** يجب أن يستطيع باحث مستقل فهم المنهج وإعادة تطبيقه.
- **تقليل الضرر:** حماية الضحايا والمصادر والبيانات الحساسة أولوية تحريرية وتقنية.
- **المساءلة:** سجل واضح للتحديثات والتصحيحات وتعارض المصالح.

### الهدف الاستراتيجي

تحويل aisearcharab.com إلى مرجع عربي متخصص في التحقيق الرقمي وأدواته، بحيث تُبنى سلطته على جودة الأدلة، وضوح المنهج، وتراكم الأصول المعرفية؛ لا على كثافة النشر وحدها.

### مؤشرات النجاح

- نسبة المواد التي تحتوي سجلاً كاملاً للمصادر.
- معدل الروابط الميتة والمصادر غير المؤرشفة.
- عدد الأدوات والمنهجيات القابلة لإعادة الاستخدام.
- زمن التصحيح بعد اكتشاف الخطأ.
- نسبة الزيارات العضوية ذات النية البحثية المتخصصة.
- عدد الإحالات الأكاديمية أو الإعلامية الموثوقة.
- Core Web Vitals الميدانية، لا المختبرية فقط.

## 2. البنية التقنية

### 2.1 النهج العام

اعتماد معمارية **Static-First** باستخدام Hugo، مع إبقاء أي وظائف ديناميكية خارج المسار الحرج للواجهة العامة.

```text
Source Content + Structured Data
          ↓
Validation / Fact / Link / Schema Checks
          ↓
Hugo Build
          ↓
Static Output
          ↓
GitHub Pages + Custom Domain + CDN/DNS Layer
```

### 2.2 المكونات الأساسية

| الطبقة | التقنية | الغرض |
|---|---|---|
| العرض | Hugo | توليد صفحات ثابتة سريعة وقابلة للأرشفة |
| المحتوى | Markdown | كتابة قابلة للمراجعة والتتبع عبر Git |
| البيانات | JSON/YAML | الكيانات، المصادر، العلاقات، وسجلات التحقق |
| الأتمتة | Python | الجمع، التحويل، التحقق، وبناء التقارير |
| CI/CD | GitHub Actions | الاختبارات، البناء، والفحص قبل النشر |
| الاستضافة | GitHub Pages | نشر ثابت منخفض التكلفة |
| النطاق | aisearcharab.com | الهوية العامة للمنصة |

### 2.3 قيود التصميم

- لا قاعدة بيانات إنتاجية ما لم توجد حالة استخدام لا يمكن حلها بشكل ثابت.
- لا JavaScript لمجرد الزينة؛ كل حزمة يجب أن تبرر أثرها.
- لا أسرار أو مفاتيح API داخل المستودع أو ملفات الواجهة.
- لا تشغيل مباشر لوكلاء البحث داخل المتصفح.
- لا دمج لمخرجات AI في المحتوى دون مراجعة بشرية موثقة.

### 2.4 الأداء

الهدف ليس وعداً مطلقاً بتحميل أقل من 0.5 ثانية لكل مستخدم، بل تقليل الكلفة التقنية وتحقيق مؤشرات أداء ميدانية قوية.

**ميزانية الأداء المقترحة لكل صفحة قياسية:**

- HTML مضغوط: أقل من 50KB.
- CSS حرج: أقل من 25KB.
- JavaScript أولي: أقل من 70KB كلما أمكن.
- الصور: AVIF/WebP مع أحجام متجاوبة.
- الخطوط: أوزان محدودة، تحميل محلي، و`font-display: swap`.
- لا طرف ثالث في المسار الحرج دون ضرورة.

## 3. الأمان والخصوصية

### 3.1 المبدأ

لا يوجد "أمان مطلق". الهدف هو تقليل السطح الهجومي، منع الأخطاء الشائعة، اكتشاف الانحرافات مبكراً، وتقليل أثر أي اختراق محتمل.

### 3.2 الضوابط الدنيا

- حماية الفرع الرئيسي ومراجعة Pull Requests.
- فحص الأسرار والاعتماديات والكود في CI.
- أقل صلاحيات ممكنة لـ GitHub Actions.
- تثبيت إصدارات Actions بواسطة SHA عند النضج.
- تفعيل Dependabot أو بديل مكافئ.
- سياسة إفصاح أمني وملف `SECURITY.md`.
- أرشفة المصادر الحساسة بصورة قانونية وآمنة.
- فصل البيانات العامة عن مواد التحقيق غير المنشورة.
- عدم رفع أدلة حساسة أو بيانات مصادر بشرية إلى مستودع عام.

### 3.3 تصنيف البيانات

| التصنيف | أمثلة | مكان التخزين |
|---|---|---|
| عام | مقالات منشورة، أدوات، مصادر مفتوحة | المستودع العام |
| داخلي | مسودات، ملاحظات تحريرية | مستودع خاص أو مساحة مقيدة |
| حساس | هويات مصادر، وثائق غير منشورة | تخزين مشفر ومحدود الوصول |
| شديد الحساسية | بيانات قد تعرض أشخاصاً للخطر | لا ترفع إلى GitHub؛ بروتوكول منفصل |

## 4. نموذج المحتوى والتحقيق

### 4.1 أنواع المحتوى

- تحقيقات رقمية.
- أدلة أدوات OSINT.
- دراسات حالة.
- قواعد بيانات عامة مدققة.
- شروحات منهجية.
- تحليلات شبكية وزمنية.
- ملفات كيانات.
- صفحات تصحيح وتحديث.

### 4.2 دورة حياة التحقيق

```text
Pitch
→ Scoping
→ Hypothesis
→ Source Map
→ Collection
→ Verification
→ Analysis
→ Right of Reply
→ Legal/Ethical Review
→ Publication
→ Monitoring
→ Correction/Update
```

### 4.3 معيار الأدلة

لكل ادعاء جوهري يجب تحديد:

- `claim_id`
- نص الادعاء
- نوعه: حقيقة / تقدير / استنتاج / ادعاء طرف ثالث
- المصدر أو مجموعة المصادر
- مستوى الثقة
- تاريخ التحقق
- اسم أو معرف المراجع
- حالة النشر
- ملاحظات التعارض أو النقص

### 4.4 مثال Front Matter

```yaml
---
title: "عنوان التحقيق"
date: 2026-06-19
lastmod: 2026-06-19
status: "reviewed"
authors:
  - "فريق aisearcharab"
reviewers:
  - "editor-id"
content_type: "investigation"
entities:
  - id: "entity-example"
claims:
  - "claim-001"
sources:
  - "source-001"
confidence: "high"
corrections: []
---
```

## 5. Provenance Tracking

### 5.1 سجل المصدر

كل مصدر يحتاج، متى أمكن، إلى الحقول التالية:

```json
{
  "id": "source-001",
  "title": "Source title",
  "publisher": "Publisher",
  "url": "https://example.org/item",
  "archive_url": "https://archive.example/item",
  "published_at": "2026-01-10T09:00:00Z",
  "accessed_at": "2026-06-19T08:00:00+03:00",
  "source_type": "official-document",
  "language": "ar",
  "reliability": "primary",
  "integrity": {
    "sha256": "optional-hash",
    "local_copy": false
  },
  "notes": "Scope and limitations"
}
```

### 5.2 قواعد الاستخدام

- المصدر الأولي مقدم على النقل الثانوي.
- لقطات الشاشة ليست بديلاً عن الرابط والنسخة المؤرشفة.
- يجب حفظ تاريخ الوصول للمحتوى المتغير.
- الادعاءات المتنازع عليها تحتاج تمثيل الأطراف ذات الصلة.
- غياب الدليل لا يُحوّل الفرضية إلى حقيقة.

## 6. الوكلاء الذكيون

### 6.1 الدور المسموح

يمكن للوكلاء تنفيذ:

- اكتشاف روابط وكيانات مرشحة.
- استخراج بيانات منظمة من مصادر عامة.
- إزالة التكرار وتوحيد الصيغ.
- إنشاء مسودات جداول زمنية أو خرائط علاقات.
- فحص الروابط والبيانات الوصفية.
- اقتراح أسئلة تحقق وثغرات بحثية.

### 6.2 الدور المحظور دون مراجعة بشرية

- تقرير صحة ادعاء نهائياً.
- تسمية شخص بارتكاب مخالفة أو جريمة.
- نشر بيانات شخصية حساسة.
- اتخاذ قرار تحريري أو قانوني.
- تحويل الارتباط الإحصائي إلى سببية.
- نشر توقعات أو درجات مخاطر بلا منهج معلن.

### 6.3 مخرجات الوكيل

كل مخرج آلي يجب أن يحتوي:

- وقت التنفيذ.
- الإصدار أو النموذج المستخدم.
- المدخلات ومصادرها.
- المعالجة المنفذة.
- الأخطاء والاستثناءات.
- درجة الثقة.
- حالة المراجعة البشرية.

## 7. الهوية البصرية وتجربة الاستخدام

### 7.1 المبادئ

- واجهة عربية RTL أولاً، مع دعم الإنجليزية لاحقاً.
- Glassmorphism محدود ووظيفي، لا على حساب التباين أو الأداء.
- هيكل Dashboard للمحتوى الكثيف، مع صفحات قراءة هادئة للتحقيقات الطويلة.
- أولوية لسهولة الوصول، وضوح الأدلة، وسرعة الانتقال بين الادعاء ومصدره.

### 7.2 المكونات الأساسية

- شريط بحث مركزي.
- بطاقات التحقيقات والمنهجيات.
- Timeline تفاعلي خفيف.
- Entity Profile.
- Source Drawer.
- Claim Evidence Matrix.
- Network Graph عند وجود قيمة تحليلية حقيقية.
- سجل تصحيحات ظاهر.

### 7.3 الوصول

- توافق WCAG 2.2 AA كهدف تنفيذي.
- تنقل كامل بلوحة المفاتيح.
- تباين واضح وعدم الاعتماد على اللون وحده.
- نصوص بديلة للصور والرسوم.
- دعم `prefers-reduced-motion`.

## 8. SEO وGEO وEntity Authority

### 8.1 الهدف

بناء كيان معرفي واضح يمكن لمحركات البحث وأنظمة الإجابة فهمه، مع عدم التضحية بالدقة لصالح الحشو أو الصياغات المصطنعة.

### 8.2 التنفيذ

- عناوين مباشرة تعكس سؤال المستخدم الحقيقي.
- ملخص تنفيذي قابل للاقتباس في بداية المادة.
- تعريف المصطلحات والكيانات بوضوح.
- روابط داخلية دلالية بين التحقيقات والأدوات والكيانات.
- صفحات مؤلفين ومراجعين ومنهجية وتصحيحات.
- ملفات Sitemap وRSS سليمة.
- Canonical URLs ثابتة.
- بيانات منظمة متوافقة مع المحتوى الظاهر.
- عدم استخدام Schema مضلل أو غير مدعوم بالمحتوى.

### 8.3 أنواع Schema المقترحة

- `Organization`
- `WebSite`
- `SearchAction`
- `Article` / `NewsArticle` بحسب المادة
- `Person`
- `Dataset`
- `SoftwareApplication` للأدوات الفعلية
- `BreadcrumbList`
- `FAQPage` فقط عند وجود أسئلة وأجوبة حقيقية ظاهرة

مصطلح "Research Entity" ليس نوع Schema مستقلاً؛ تُنمذج المؤسسة بتركيب أنواع وخصائص صحيحة بدلاً من اختراع نوع غير معتمد.

## 10. خارطة التنفيذ

### المرحلة 0 — الحوكمة

- تثبيت الوثائق المرجعية.
- تحديد الترخيص والسياسة التحريرية.
- إعداد `CODEOWNERS`, `SECURITY.md`, وسياسة التصحيحات.

### المرحلة 1 — الأساس التقني

- توليد Hugo Skeleton.
- إعداد RTL والثيم الأساسي.
- إعداد GitHub Actions للبناء والفحص والنشر.
- ربط النطاق وشهادة TLS.

### المرحلة 2 — نموذج البيانات

- تعريف Schemas للمصادر والادعاءات والكيانات.
- إضافة Validators واختبارات.
- بناء صفحة مصدر وصفحة كيان.

### المرحلة 3 — MVP تحريري

- الصفحة الرئيسية.
- صفحة تحقيق.
- صفحة Toolkit.
- صفحة منهجية.
- سجل تصحيحات.
- بحث ثابت أو مفهرس خارجياً عند الضرورة.

### المرحلة 4 — الأتمتة

- وكلاء محليون لجمع المصادر العامة.
- Deduplication وEntity Resolution.
- توليد تقارير تحقق أولية.
- حفظ سجلات التشغيل والمراجعة.

### المرحلة 5 — التوسع

- قواعد بيانات عامة قابلة للتنزيل.
- خرائط زمنية وشبكية محسوبة مسبقاً.
- واجهة متعددة اللغات.
- شراكات بحثية وتحريرية.

## 12. قرارات معمارية محسومة

- **Hugo بدلاً من CMS ديناميكي** في الإصدار الأول.
- **Markdown/JSON/YAML بدلاً من قاعدة بيانات** للمحتوى العام الأساسي.
- **GitHub Actions بدلاً من الاعتماد على بيئة تطوير تفاعلية** للأتمتة المتكررة.
- **Human-in-the-loop إلزامي** لكل نتيجة تحريرية حساسة.
- **الأداء والأمان يقاسان باختبارات ومؤشرات**، لا بصياغات مطلقة.
- **Provenance Tracking جزء من نموذج البيانات** وليس ملحقاً تحريرياً اختيارياً.
